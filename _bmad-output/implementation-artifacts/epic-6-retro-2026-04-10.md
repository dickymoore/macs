# Epic 6 Retrospective

Date: 2026-04-10
Epic: 6
Epic Title: Preserve Auditability and Governance Boundaries
Status: complete

## Epic Review

Epic 6 finished with all four stories complete and accepted:

- `6.1` persisted the durable event trail and inspectable history readers.
- `6.2` preserved intervention rationale and causation across recovery and reassignment.
- `6.3` enforced explicit decision-rights classes and guarded operator actions.
- `6.4` added repo-local governance policy, governed-surface routing controls, and audit-content redaction or omission.

The epic landed in the right order. Durable event and history seams came first, which let later rationale, decision-rights, and governance work extend existing controller-owned paths instead of inventing parallel storage or approval systems.

## What Worked

- Reusing the existing `events` store and inspect surfaces kept auditability changes incremental and easy to regression-test.
- Keeping policy work in `tools/orchestration/policy.py` avoided fragmented authority and made Epic 6 feel like hardening, not subsystem churn.
- Extending existing CLI regression modules instead of creating new test silos kept red-green slices tight and made failures easier to localize.
- Explicit BMAD QA acceptance passes kept catching final read-side and human-readable gaps before stories were marked done.

## Friction and Risks

- Human-readable output repeatedly lagged behind JSON or state-layer changes. Read-side QA needs to stay explicit, not assumed.
- Repo-local behavior is now spread across bootstrap files, policy files, tmux metadata files, and legacy targeting conventions. That is the main adoption risk entering Epic 7.
- Governance defaults are now real, but the configuration model is still implicit. Operators can edit files directly, yet MACS does not present the full config domain set as one coherent, inspectable surface.

## Follow-Through From Previous Retrospectives

No prior retrospective artifact was found for Epic 5, so there were no formal carry-forward action items to audit.

## Next Epic Preparation

Epic 7 should preserve the controller-owned authority boundary while making adoption and configuration explicit:

- Story 7.1 should bootstrap and expose separate repo-local config domains for controller defaults, adapter settings, routing policy, governance policy, and repo-local state paths without moving authority out of controller code.
- Story 7.2 should build the real setup and validation flow on top of those inspectable config surfaces instead of relying on documentation-only guidance.
- Story 7.3 should treat `.codex/tmux-session.txt`, `.codex/tmux-socket.txt`, `.codex/target-pane.txt`, and legacy `tools/tmux_bridge/target_pane.txt` compatibility as a release-evidence concern, not an assumption.
- Story 7.4 should align contributor-facing adapter guidance with the actual contract, degraded-mode expectations, and qualification evidence already embedded in the codebase.

- Keep Epic 7 work on existing controller seams first: `session.py`, `policy.py`, `cli/main.py`, `tools/tmux_bridge/start_worker.sh`, and current setup tests.
