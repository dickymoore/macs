# Epic 7 Retrospective

Date: 2026-04-10
Epic: 7
Epic Title: Adopt, Configure, and Migrate MACS in Real Repositories
Status: complete

## Epic Review

Epic 7 finished with all four stories complete and accepted:

- `7.1` separated repo-local controller, adapter, routing, governance, and state configuration domains.
- `7.2` added `macs setup validate` and `macs setup dry-run` as controller-owned onboarding surfaces.
- `7.3` documented and regression-locked single-worker compatibility boundaries and legacy metadata readability.
- `7.4` published contributor-facing adapter guidance and surfaced the shared adapter contract directly on `macs adapter inspect|validate`.

The epic sequencing held up. Configuration separation came first, then onboarding, then migration compatibility, then contributor guidance. That order let later documentation and validation work reuse real controller seams instead of inventing tutorial-only workflows.

## What Worked

- Reusing `tools/orchestration/setup.py`, `config.py`, and the existing setup command family kept adoption work incremental and auditable.
- Shipping docs and regression coverage inside each story prevented Epic 7 from becoming a “write docs later” cleanup project.
- The explicit BMAD QA acceptance pass kept finding the final read-side gaps that broad green test runs would not have caught on their own.
- Contributor guidance landed cleanly once the shared adapter contract metadata was centralized in one seam instead of being duplicated across docs and tests.

## Friction and Risks

- Human-readable output kept lagging behind JSON and state-layer changes. Read-side acceptance still needs to be treated as a first-class risk.
- Setup, compatibility, and contributor guidance now exist, but the deterministic test architecture Epic 8 needs is still spread across broad CLI suites instead of dedicated unit and contract layers.
- The current validation surface is credible for feature delivery, but release-readiness proof will be harder to maintain if contract checks and failure drills remain mixed together in large end-to-end modules.

## Follow-Through From Previous Retrospectives

- Story `8.1` should carve out deterministic unit and contract suites for controller invariants and adapter qualification checks, using current helper seams instead of burying more assertions in the broad CLI tests.
- Story `8.2` should keep tmux-backed integration and failure drills separate from the deterministic unit layer so catastrophe-grade scenarios stay isolated and repeatable.
- Story `8.3` should build the four-worker dogfood scenario on the same governed runtime mix and inspection surfaces Epic 7 already documented.
- Story `8.4` should aggregate existing validation evidence into a release-gate summary rather than inventing a second source of truth.
